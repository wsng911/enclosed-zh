# enclosed-zh Prompts

> 项目：CorentinTh/enclosed 汉化版（enclosed-zh）
> 技术栈：SolidJS + Vite 前端，Hono + Node.js 后端，端对端加密，pnpm monorepo

---

## 功能迭代

**1. 添加阅后即焚确认页面**
在 enclosed-zh 中，当用户打开一条设置为"阅后即焚"的消息时，目前直接显示内容。加个一个确认步骤：先显示警告页面提示"此消息查看后将被永久删除"，用户点击确认后才显示内容，防止误触导致消息丢失。

**2. 支持富文本消息格式**
在 enclosed-zh 的消息创建页面，添加 Markdown 编辑器支持。用户可以使用 Markdown 语法编写格式化消息（粗体、列表、代码块等），接收方查看时渲染为 HTML，同时保持端对端加密。

**3. 添加消息发送统计**
在 enclosed-zh 的管理后台添加统计面板，展示：今日创建消息数、今日查看消息数、当前存储的消息总数、过期消息清理记录。数据从服务端日志聚合，不涉及消息内容。

**4. 支持自定义消息过期时间**
在 enclosed-zh 的消息创建表单中，将过期时间选项从预设列表扩展为支持自定义输入。用户可以输入具体的小时数（1-720小时），后端验证范围合法性后存储。

**5. 添加消息访问密码强度提示**
在 enclosed-zh 设置消息密码时，添加实时密码强度指示器（弱/中/强），并在密码过弱时显示警告提示，建议用户使用更强的密码保护敏感消息。

## Bug 修复

**6. 修复消息链接在微信内置浏览器中无法打开的问题**
在 enclosed-zh 中，分享的消息链接在微信内置浏览器中打开时，由于 WebCrypto API 的限制，解密失败并显示错误。请检测微信浏览器环境，提示用户在外部浏览器中打开链接。

**7. 修复大文件附件加密时内存溢出**
在 enclosed-zh 中，当用户附加超过 50MB 的文件时，前端加密过程会导致浏览器内存溢出。把大文件加密改为流式处理，分块加密后再合并，避免一次性将整个文件加载到内存。

**8. 修复消息过期时间显示时区错误**
在 enclosed-zh 中，消息的过期时间在创建者和接收者的界面上显示不一致，原因是服务端存储 UTC 时间但前端未正确转换为本地时区。请统一使用 `Intl.DateTimeFormat` 进行时区转换显示。

**9. 修复并发创建消息时 ID 碰撞**
在 enclosed-zh 的高并发场景下，同时创建多条消息时偶发 ID 碰撞导致消息覆盖。检查消息 ID 生成算法，确保使用足够长度的随机 ID（至少 128 位），并在数据库层添加唯一性约束。

**10. 修复语言切换后页面部分文本未更新**
在 enclosed-zh 中，通过语言切换器切换语言后，页面中部分动态生成的文本（如错误提示、Toast 通知）仍显示旧语言。检查这些文本是否正确使用了 `useI18n()` 的响应式 `t()` 函数。

## 重构

**11. 将加密逻辑提取为独立的 Web Worker**
enclosed-zh 的前端加密/解密操作在主线程执行，对于大消息会导致 UI 卡顿。把 `@enclosed/crypto` 包的加密操作迁移到 Web Worker 中执行，通过 `postMessage` 与主线程通信，保持 UI 响应流畅。

**12. 统一前后端的错误码定义**
enclosed-zh 的前端和后端各自定义了错误处理逻辑，错误码不统一。在 `@enclosed/lib` 包中定义共享的错误码枚举，前后端统一引用，确保错误信息的一致性。
